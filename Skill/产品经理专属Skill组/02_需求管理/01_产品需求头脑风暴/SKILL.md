# 需求头脑风暴与方案设计

## 概述

通过自然的协作对话，将想法转化为完整的设计和规格说明。

首先了解当前项目背景，然后逐个提问来细化想法。当你理解了要构建的内容后，分小节展示设计（200-300字），每部分后确认是否正确。

## 工作流程

**理解想法：**
- 首先查看当前项目状态（文件、文档、最近提交）
- 逐个提问以细化想法
- **反馈工具**: 如果有 `mcp-feedback-enhanced`（如 `ask_followup_question`），请务必使用它来提问。如果没有，使用标准对话。
- 尽可能使用选择题，开放式问题也可以
- 每条消息只问一个问题——如果某个主题需要深入探讨，拆分成多个问题
- 重点理解：目标、约束、成功标准

**文档与规格说明：**
- **PRD (需求规格说明书)**: 编写标准化的需求文档，包含背景、用户流程、功能详细说明、验收标准。
- **项目介绍 (Project Introduction)**: 编写面向利益相关者或用户的项目愿景、核心价值、Roadmap。
- **特性说明 (Feature Docs)**: 针对具体模块编写交互细节与逻辑说明。

**探索方案：**
- 提出2-3种不同方案并说明权衡
- 以对话方式展示选项，给出你的推荐和理由
- 优先展示推荐方案并说明原因

**展示设计：**
- 确认理解需求后，展示设计方案
- 拆分成200-300字的小节
- 每部分结束后询问是否正确
- 涵盖：架构、组件、数据流、错误处理、测试
- 如果有不合理的地方，准备回溯和澄清

## 设计完成后

**文档：**
- 将确认后的设计写入 `docs/plans/YYYY-MM-DD-<主题>-design.md`
- 如果可用，使用简洁写作技能
- 将设计文档提交到 git

**实现（如果继续）：**
- 询问："准备好开始实现了吗？"
- 使用 git 工作区创建隔离工作空间
- 使用计划编写功能创建详细实现方案

## 核心原则

- **一次一个问题** - 不要同时问多个问题造成压力
- **优先选择题** - 比开放式问题更容易回答
- **坚决精简** - 从所有设计中移除不必要的功能
- **探索备选方案** - 确定前总是提出2-3种方法
- **增量验证** - 分部分展示设计，逐个验证
- **保持灵活** - 有问题时随时回溯澄清
